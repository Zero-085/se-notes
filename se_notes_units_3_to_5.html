Basic Concepts of Software Design
▼

Software design is the process of transforming requirements into a blueprint for construction. It acts as a bridge between requirements and code.

Two levels of design

  • Architectural Design (High Level): Defines overall structure — subsystems, components, how they interact.
  • Low Level Design (Detailed): Defines internal logic of each module — algorithms, data structures.

Design goals

  • Correctness, efficiency, understandability, maintainability, reusability
Architectural Design
▼

Defines the overall structure of the software system. Identifies major components and their relationships.

Common architectural styles

  • Layered: Each layer serves the one above it (e.g. OSI model)
  • Client-Server: Clients request, servers respond
  • Pipe and Filter: Data flows through processing stages
  • Repository: Shared data store accessed by multiple components
  • Event-Driven: Components react to events
  • MVC (Model-View-Controller): Separates data, UI, and logic
Low Level Design — Modularization, DSC, Pseudo Code, Flowcharts
▼

Modularization

Breaking the system into smaller, manageable modules. Each module handles a specific function. Key principles: single responsibility, information hiding.

Design Structure Charts (DSC)

Hierarchical diagram showing modules and their calling relationships. Shows which module calls which — top-down view of system. Key elements: rectangles (modules), arrows (calls), diamonds (conditions), loops (repetition).

Pseudo Code

Informal, human-readable description of an algorithm — not actual code. Uses plain English with programming constructs (if, while, for, etc.).

BEGIN READ number IF number MOD 2 == 0 THEN PRINT "Even" ELSE PRINT "Odd" END

Flowcharts

Graphical representation of an algorithm using standard symbols. Below is the visual representation of the Even/Odd logic:

flowchart TD A([BEGIN]) --> B[/READ number/] B --> C{number MOD 2 == 0?} C -- Yes --> D[/PRINT Even/] C -- No --> E[/PRINT Odd/] D --> F([END]) E --> F
  • Oval — Start/End
  • Rectangle — Process
  • Diamond — Decision
  • Parallelogram — Input/Output
  • Arrow — Flow of control

Design Strategies
▼

Function Oriented Design (FOD)

  • System viewed as a set of functions/processes
  • Uses DFDs (Data Flow Diagrams) to model
  • Focus: what the system does
  • Top-down decomposition of functions

Object Oriented Design (OOD)

  • System viewed as a collection of interacting objects
  • Objects have attributes (data) and methods (behavior)
  • Concepts: encapsulation, inheritance, polymorphism, abstraction
  • Uses UML diagrams (class diagrams, sequence diagrams)

Top-Down vs Bottom-Up Design

Top-DownBottom-Up
Start from overall system, break into partsStart with basic modules, combine upward
Good for well-understood problemsGood for reusing existing components
Uses stubs for testingUses drivers for testing
Early visibility of system structureComponents tested early
Software Metrics — Halstead, FP, Cyclomatic Complexity
▼

Halstead's Software Science

Measures complexity based on operators and operands in source code.

  • n1 = number of distinct operators
  • n2 = number of distinct operands
  • N1 = total occurrences of operators
  • N2 = total occurrences of operands
Vocabulary (n) = n1 + n2 Program Length (N) = N1 + N2 Volume (V) = N × log₂(n) Difficulty (D) = (n1/2) × (N2/n2) Effort (E) = D × V
Effort (E) is most important — it estimates mental effort to code or understand the program.

Function Point (FP) Based Measures

Measures functionality from the user's perspective. Based on 5 components:

  • External Inputs (EI)
  • External Outputs (EO)
  • External Inquiries (EQ)
  • Internal Logical Files (ILF)
  • External Interface Files (EIF)
UFP = Σ(count × complexity weight) FP = UFP × (0.65 + 0.01 × Σ Fi) where Fi = 14 general system characteristics (GSC), each rated 0–5
FP is language-independent. Used for cost/effort estimation early in the project.

Cyclomatic Complexity (McCabe)

Measures the number of independent paths through source code. Higher = more complex = harder to test.

V(G) = E - N + 2P where E = edges, N = nodes, P = connected components (usually 1) Also: V(G) = number of decision points + 1 (Decision points: if, while, for, case, &&, ||)
V(G) ValueRisk
1–10Simple, low risk
11–20Moderate
21–50High risk
>50Very high risk, untestable
Control Flow Graph (CFG): Nodes = statements/blocks, Edges = control flow between them. V(G) = minimum number of test cases needed.
Testing Objectives & Types
▼

Goals of testing

  • Find defects before delivery
  • Verify software meets requirements
  • Validate software does what the user actually needs
  • Testing shows presence of bugs, not their absence (Dijkstra)

Levels of testing

TypeWhoPurpose
Unit TestingDeveloperTest individual modules/functions in isolation
Integration TestingDeveloper/TesterTest interaction between combined modules
System TestingTesterTest the complete system as a whole
Acceptance TestingClient/UserValidate system meets business requirements

Regression Testing

Re-running tests after code changes to ensure existing functionality is not broken. Done after every bug fix or new feature addition.

Performance Testing

Tests system under load — speed, scalability, stability. Includes load testing, stress testing, and spike testing.

Top-Down & Bottom-Up Testing Strategies
▼

Test Stubs

Dummy modules that simulate a called module that isn't developed yet. Used in top-down testing — you test higher-level modules first, stubs replace lower ones.

Test Drivers

Dummy modules that simulate the calling module. Used in bottom-up testing — you test lower-level modules first, drivers call them.

Top-Down TestingBottom-Up Testing
DirectionMain module first, downwardLeaf modules first, upward
NeedsStubsDrivers
AdvantageEarly system behavior visibleReal modules tested early
DisadvantageWriting stubs is tediousSystem-level errors found late
White Box Testing (Structural)
▼

Tests internal structure/logic of code. Tester has full access to source code. Also called glass-box or clear-box testing.

Coverage criteria

  • Statement Coverage: Every statement executed at least once
  • Branch Coverage: Every branch (true/false) of decisions covered
  • Path Coverage: Every possible path through the program covered
  • Condition Coverage: Every condition evaluated to both true and false

Techniques

  • Basis Path Testing — uses cyclomatic complexity to determine test cases
  • Loop Testing — test loops for zero, one, typical, max, min iterations
  • Data Flow Testing — track variable definitions and uses
Key point: White box testing is done by developers. It ensures code correctness, not user requirement fulfillment.
Black Box Testing (Functional)
▼

Tests functionality without knowledge of internal code. Based entirely on requirements/specifications. Done by testers, not developers.

Techniques

  • Equivalence Partitioning: Divide inputs into valid/invalid classes; test one value from each class
  • Boundary Value Analysis (BVA): Test at boundaries (min, min+1, max-1, max). Bugs cluster at boundaries.
  • Decision Table Testing: Tabulate conditions and actions for complex logic
  • State Transition Testing: Test system behavior in different states
  • Use Case Testing: Test based on use case scenarios
Example — BVA: For a field accepting 1–100, test values: 0, 1, 2, 99, 100, 101

Static Testing — Reviews, Walkthroughs, Inspections
▼

Static testing doesn't execute code. It reviews artifacts (code, design, docs) manually to find defects early.

Formal Technical Review (FTR / Peer Review)

  • Structured meeting where team reviews software product
  • Roles: moderator, author, reviewers, recorder
  • Produces a review report with defects listed
  • Author cannot moderate their own work

Walkthrough

  • Author walks the team through the document/code
  • Informal — no strict rules
  • Goal: understanding + finding defects
  • Author leads the discussion

Code Inspection

  • Most formal type of review
  • Uses checklists to systematically find defects
  • Moderator leads (not the author)
  • Defects logged; author must fix and re-inspect if needed
WalkthroughInspection
FormalityLowHigh
LeaderAuthorModerator (not author)
ChecklistNoYes
Follow-upNot mandatoryMandatory

Compliance with Standards

Code/design must follow organizational coding standards and design conventions — naming, formatting, commenting, error handling, etc. Checked during reviews/inspections.

Software Maintenance
▼

Software as an evolutionary entity

Software must evolve to stay useful. Lehman's Laws say software either continues to adapt or becomes progressively less useful.

Need for maintenance

  • Bugs found post-delivery
  • Environment changes (OS, hardware, regulations)
  • User requirements change
  • Performance improvements needed

Categories of maintenance

TypePurposeExample
CorrectiveFix bugs found after deliveryFixing a crash on login
AdaptiveAdapt to new environmentUpgrade for new OS version
PerfectiveEnhance functionality/performanceAdding a new feature
PreventivePrevent future failures (re-engineering)Refactoring messy code
Cost fact: Maintenance consumes 60–80% of total software lifecycle cost. Preventive maintenance reduces long-term cost.
Re-Engineering & Reverse Engineering
▼

Software Re-Engineering

Rebuilding or restructuring existing software to improve quality without changing its functionality.

flowchart LR A(Inventory Analysis) --> B(Document Restructuring) B --> C(Reverse Engineering) C --> D(Code Restructuring) D --> E(Data Restructuring) E --> F(Forward Engineering)
  • Steps: Inventory analysis → Document restructuring → Reverse engineering → Code restructuring → Data restructuring → Forward engineering

Reverse Engineering

Analyzing existing software to understand its design and requirements — going backward from code to design/specs.

  • Used when documentation is missing or outdated
  • Produces design documents, models, specs from code
  • Component of re-engineering process
Difference: Reverse engineering = understand existing system. Re-engineering = understand + rebuild/improve it.
Software Configuration Management (SCM)
▼

SCM manages changes to software throughout its lifecycle. Ensures integrity, traceability, and control.

SCM activities

  • Configuration Identification: Identify items to be controlled (SCIs — Software Configuration Items)
  • Change Control: Formal process to request, evaluate, approve/reject changes
  • Version Control: Track different versions of software artifacts (Git, SVN)
  • Configuration Audit: Verify that configuration items conform to requirements
  • Status Reporting: Record and report status of change requests

Change Control Process

flowchart LR A([Change Request]) --> B(Evaluate by CCB) B --> C{Decision} C -- Approve --> D(Implement & Test) D --> E([Release]) C -- Reject --> F([Close Request])
  • Change request submitted → Evaluated by CCB (Change Control Board) → Approved/Rejected → Implemented → Tested → Released

Software Version Control

  • Tracks all changes to code with timestamps and author info
  • Allows rollback to previous versions
  • Supports branching and merging
  • Tools: Git, SVN, Mercurial

CASE Tools (Computer Aided Software Engineering)

Software tools that support software engineering activities:

  • Upper CASE: Requirements, planning, modeling (draw.io, Enterprise Architect)
  • Lower CASE: Code generation, testing, debugging (IDEs, compilers)
  • Integrated CASE: Cover full lifecycle
Project Estimation — Cost, Effort, Schedule
▼

Parameters to estimate

  • Size: LOC, Function Points
  • Effort: Person-months required
  • Duration: Calendar time (schedule)
  • Cost: Money required

COCOMO (Constructive Cost Model) — Barry Boehm

Algorithmic model to estimate effort and duration based on estimated LOC.

Basic COCOMO: Effort (E) = a × (KLOC)^b [person-months] Duration (D) = c × (E)^d [months] Organic mode: E = 2.4 × (KLOC)^1.05, D = 2.5 × E^0.38 Semi-detached: E = 3.0 × (KLOC)^1.12, D = 2.5 × E^0.35 Embedded mode: E = 3.6 × (KLOC)^1.20, D = 2.5 × E^0.32
ModeDescription
OrganicSmall, simple projects. Experienced team. Well-understood requirements.
Semi-detachedMedium complexity. Mixed experience team.
EmbeddedComplex, tight constraints. Real-time or safety-critical systems.
Intermediate COCOMO adds 15 cost drivers (product, hardware, personnel, project attributes) as multipliers to the basic estimate.
Resource Allocation Models
▼

Putnam Model (Norden-Rayleigh Curve)

Models staffing over time as a bell-shaped curve (Rayleigh curve). Peak staffing is at about 40% of the project timeline.

L = C × K^(1/3) × T^(4/3) where L = LOC, K = effort (person-years), T = development time, C = technology constant

Resource Allocation

  • Identify tasks, estimate effort per task
  • Assign people based on skills and availability
  • Monitor and rebalance as project progresses
  • Don't add people late to a late project (Brooks's Law: adds more delay)
Software Risk Analysis & Management
▼

Risk

A potential problem that may or may not occur. Characterized by probability and impact.

Risk Exposure = Probability × Impact

Types of risk

  • Project risk: Schedule delays, budget overruns, staff issues
  • Technical risk: Design problems, technology failures
  • Business risk: Market changes, competitor products

Risk management steps

  • Risk Identification: List all possible risks (checklists, brainstorming)
  • Risk Analysis: Estimate probability and impact for each risk
  • Risk Prioritization: Rank by risk exposure
  • Risk Planning: Define mitigation or contingency strategies
  • Risk Monitoring: Track risks throughout the project

Risk strategies

  • Avoidance: Redesign to eliminate the risk
  • Mitigation: Reduce probability or impact
  • Transfer: Shift risk to third party (insurance, outsourcing)
  • Acceptance: Accept and deal with consequences if it occurs
